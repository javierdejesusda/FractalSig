# FractalSig Triage Report — 2026-05-03

This document captures the state of the repository at the start of the publication-quality
rewrite. Each finding lists: claim/assumption, current behavior, evidence (file:line), and
the fix status as of this commit.

The companion implementation plan lives in `docs/superpowers/plans/` (gitignored).

---

## What works

### F1. Imports and GPU
The `fractalsig` conda env is healthy. `torch 2.5.1+cu124` reports `cuda: True`,
`jax 0.4.38` enumerates `[CudaDevice(id=0)]`, `pywt`, `ptwt`, `iisignature`, `fbm`, and
`hydra` all import cleanly.

### F2. fBM data generation
`fractalsig/data_gen.py:40-118` generates fBM paths with H ≈ 0.1 using FBM library. The
Davies-Harte → Cholesky fallback at line 96-102 correctly handles negative-eigenvalue
failures. The on-disk dataset `data/rough_volatility.npy` has the expected shape
`(5000, 256, 2)` (paths × time × [time-channel, value]).

### F3. FractalDecoder forward + backward
`fractalsig/decoder.py:201-247` produces correctly shaped output `(batch, seq_len, 1)`,
gradients flow through `ptwt.waverec`. The shape inference (`_infer_coefficient_shapes`,
lines 109-140) successfully handles arbitrary wavelet families and decomposition levels.

### F4. Existing checkpoints load and run
`checkpoints/fractal_production/fractal_decoder_best.pth` (2.5 MB) loads and produces
output. `SigDiffusions/model_checkpoints/fractal_production` (87 KB JAX checkpoint) exists.

## What is broken or unsubstantiated

### B1. Headline "70× improvement vs SigDiffusions" claim is unreproducible
**README:191** states `FractalSig: 0.985` increment-std relative error, vs
`SigDiffusions: 0.142`, with ground truth normalized to `1.000`.

**Reproduced reality** (this triage):

| Measurement | Value |
|---|---|
| Ground-truth fBM increment std (per-channel z-scored data, value channel only) | **0.629** |
| Saved `results/fractal_production_final_paths.npy` increment std | **0.453** |
| Ratio fake/real (saved sample paths) | **0.72** (under-rough by 28%) |
| Encode-decode round-trip on 100 GT paths through trained FractalDecoder | inc-std **1.295** → ratio **2.06** (over-rough by 2x) |

**Status**: Open. Neither the 0.985 number nor the 70× ratio is reproducible from the
saved artifacts. The README's metric is undocumented (which normalization, which sample
set). This is the central empirical credibility issue the publication-quality rewrite must
fix by (a) implementing baselines that actually run, (b) defining metrics with citations,
(c) running the comparison with multiple seeds and CI.

### B2. `compute_signatures` and `invert_signatures` had divergent signature layouts
`SigDiffusions/compute_signatures.py:90` was patched to drop the `+3` from
`iisignature.prepare(dim+3, ...)` to skip sin/cos augmentation, but
`SigDiffusions/invert_signatures.py:78` still used the un-patched `dim+3` layout. Any call
to the inversion path produced garbage or crashed.

**Status**: **Fixed** in commit `8024d49`. Both files now use
`iisignature.prepare(dim, sig_depth, "COSX2")`. Test added at
`tests/test_invert_sigs.py`.

### B3. `mode=auto` re-trains JAX every run because checkpoint name has wrong extension
`main.py:233` checks for `SigDiffusions/model_checkpoints/{cfg.experiment}.pkl`, but
`SigDiffusions/train.py:93` saves the model as `f"{folder}/{name}"` (no `.pkl` extension).
The check always returns False, so `mode=auto` re-runs JAX training every time, costing
~30-60 minutes on the laptop for no reason.

**Status**: Open. Will be fixed in Phase 2 (refactor) when the runner is rewritten.
Workaround: invoke individual modes (`mode=sample`) instead of `mode=auto`.

### B4. JAX-PyTorch signature dimension is band-aided, not aligned
`main.py:431-441` truncates or pads JAX-generated signatures to match the decoder's
expected dimension when they disagree. This silently corrupts inputs to the decoder.
Specifically, the existing checkpoint expects `logsig_dim=41` (`sig_depth=7` per
`normalization_stats.json`), but the JAX config injection sets `sig_depth=7` for
`dim=2 → expected_sig_dim=254` per `inject_jax_config` at `main.py:144-150`. The truncation
silently drops 213 dimensions, throwing away most of the signature.

**Status**: Open. Will be fixed in Phase 2 by making the decoder + JAX config share a
single source of truth for `sig_depth`.

### B5. PyTorch decoder training has no validation split
`fractalsig/train_decoder.py:308-374` only logs train loss and saves the best checkpoint
based on **train** loss. `RoughPathDataset` constructs the entire dataset and uses it for
training only; no val/test split exists.

**Status**: Open. Phase 4 fixes this.

### B6. Multi-channel decoder ignores cross-channel correlation
`fractalsig/decoder.py:219-230`: for `out_channels > 1`, the code re-runs the MLP and
slices its output per channel. There is no parameter that learns cross-channel structure;
the channels are produced independently from the same shared MLP.

**Status**: Open. Phase 4 fixes this with a single-pass channel-coupled head.

### B7. Wavelet loss is unweighted MSE
`fractalsig/train_decoder.py:331`: `criterion = nn.MSELoss()` applied to the flattened
coefficient vector treats coarse approximation coefficients (large magnitude) and detail
coefficients (small magnitude, particularly at high levels) with equal weight. The
high-frequency content the project specifically claims to recover is therefore
under-emphasized in the loss.

**Status**: Open. Phase 4 introduces `ScaleWeightedMSE`.

### B8. No tests, no CI
No `tests/` directory existed before this commit. No `.github/workflows/`. Validation was
informal (Jupyter notebooks + `scripts/verify_*.py`).

**Status**: **Partially fixed**. `tests/` now exists (one test file, `test_invert_sigs.py`).
Phase 1 adds the full smoke suite, conftest fixtures, and a CI workflow.

### B9. Single seed, no error bars, no significance tests
The README's table reports a single number per method, with no run-to-run variance,
confidence interval, or paired test. `train_decoder.py:415` accepts `--seed` but no
sweep harness varies it.

**Status**: Open. Phase 7 + Phase 9 address this via 3-seed sweeps and bootstrap CI.

### B10. Single dataset (synthetic fBM only)
The repo ships exactly one dataset: synthetic fBM with H=0.1, n=5000, seq_len=256. There
are no real-world benchmarks (financial, EEG, turbulence, audio).

**Status**: Open. Phase 3 adds 4 more datasets (S&P 500 intraday vol, Burgers turbulence,
CHB-MIT EEG, ESC-50 audio).

### B11. Dependency drift: `pywt` installed is 1.8.0 but pinned at 1.9.0
`requirements.txt:24` pins `pywavelets==1.9.0`, but the live env has `1.8.0`.

**Status**: Cosmetic. `pywt` 1.8.0 vs 1.9.0 has no observable behavior difference for
our usage. Will be tracked when CI environment is built.

## What was unsubstantiated in the README that needs revisiting

### U1. "70× improvement"
See B1. Once Phase 7 produces real cross-baseline results with CIs, the README's main
table will be replaced with the auto-generated one from `paper/tables/main_results.tex`.

### U2. "Hybrid 'Soft-Fork' architecture"
The architecture is not "hybrid" in any deep sense — it is a 1-way handoff: JAX trains a
diffusion in signature space, generates signatures, dumps them to a `.npy` file, then the
PyTorch decoder loads them. This is fine, but the README mythologizes the integration.
Will be re-described accurately in the paper Method section.

### U3. "Mathematical proof that wavelets work in Besov space"
The README hand-waves "We prove that..." but no proof or theorem statement exists in the
repo or the paper folder (which doesn't exist yet). The publication-quality plan
explicitly elects a **light-theory** path: cite Daubechies / Mallat / Triebel for the
relevant Besov-wavelet classical results, no original theorems.

---

## Summary

The repo is a working proof-of-concept that demonstrates a defensible mathematical
intuition (wavelets > Fourier for rough paths). The decoder code is clean and
reproduces a correct forward / backward pass. The scientific claim, however, is currently
neither formally proven nor empirically substantiated: the central comparison was
silently broken, the metrics are undocumented, the dataset is synthetic-only, and there
is no statistical rigor.

The 11-phase plan addresses every item above.
